# План разработки: «Договора» — SaaS для работы с договорами через ИИ

## Контекст

Greenfield SaaS-продукт. Дизайн-хэндоф готов (high-fidelity, 20 экранов).
Задача — реализовать продукт пиксель-в-пиксель по макету в правильном стеке.

**Ключевые решения:**
- DaData / ЕГРЮЛ — **нет в MVP**, все реквизиты вводятся вручную
- OAuth (Госуслуги / Яндекс ID) — **нет в MVP**, но архитектура auth должна поддерживать добавление провайдеров
- AI-провайдер — **абстракция** (интерфейс один, реализации для Claude и OpenAI)
- Реальный платёжный шлюз — **нет в MVP**, только внутренний баланс с ручным пополнением администратором
- Хостинг — **TBD**, код пишется без жёсткой привязки

## Стек

| Слой | Технология | Причина |
|------|-----------|---------|
| Frontend | Next.js 14 (App Router) + TypeScript | SSR, middleware auth, экосистема |
| Стили | Tailwind CSS | Токены из `tokens.css` → `tailwind.config` |
| UI-примитивы | shadcn/ui (headless) | Кастомизируются под дизайн без борьбы |
| Состояние | Zustand (global) + TanStack Query (server) | Минимально, без Redux |
| Формы | React Hook Form + Zod | Валидация ИНН, БИК, расчётного счёта |
| Backend | Fastify + TypeScript | Быстрее Express, WebSocket, type-safe |
| ORM | Prisma | Миграции, типизация, читаемые схемы |
| БД | PostgreSQL | ACID для транзакций, версионирование |
| Очереди | BullMQ + Redis | Фоновая генерация AI, конвертация файлов |
| Файлы | S3-совместимое (конкретный провайдер TBD) | Файлы никогда не в БД |
| AI | Абстрактный `AIProvider` interface | Подключаем Claude / OpenAI без рефакторинга |

## Схема базы данных (ключевые модели)

```
User
  id, email, passwordHash, createdAt
  → profiles: Profile[]       (ИП / ООО / физлицо — реквизиты пользователя)
  → wallet: Wallet             (баланс)

Wallet
  id, userId, balance (Decimal)
  → transactions: Transaction[]

Transaction
  id, walletId, type (CREDIT | DEBIT), amount, description,
  relatedVersionId?, createdAt

Counterparty
  id, userId, name, inn, kpp, ogrn, legalAddress, ...
  → bankDetails: BankDetail[]
  → signatories: Signatory[]
  → documents: Document[]

Signatory
  id, counterpartyId, fullName, signatureName, position,
  basisType (CHARTER | POA | ...), poaNumber?, poaDate?, poaExpiry?,
  poaFilePath?,
  scopes: SignatoryScope[]  (CONTRACT | APPENDIX | AMENDMENT | ACT)

Document
  id, userId, counterpartyId, title, number, type (CONTRACT | APPENDIX | AMENDMENT)
  → versions: Version[]

Version
  id, documentId, number (1,2,3...), status (DRAFT|IN_PROGRESS|REVIEW|APPROVED|PAID)
  filePath (S3), fileSize,
  aiSettings: Json  { protectionLevel, targetSize, customInstruction }
  createdAt
  → chatMessages: ChatMessage[]
  → purchase: Purchase?
  — APPEND-ONLY, никогда не обновляется текст/файл

ChatMessage
  id, versionId, role (USER | AI | WARNING), content, createdAt

Purchase
  id, versionId, amount, purchasedAt
  — идемпотентна, создаётся один раз

StorageQuota
  id, userId, usedBytes, limitBytes, plan (STARTER | BUSINESS | BUREAU)
```

## Правила которые нельзя нарушать

1. **Версия — append-only.** Никогда не делать `UPDATE` на текст/файл версии
2. **Файлы только в S3.** В БД — только путь
3. **AI в очереди.** Тяжёлые AI-запросы всегда через BullMQ, не в HTTP-handler
4. **Покупка — транзакция БД.** `SELECT FOR UPDATE`, идемпотентность
5. **AI-чат — постоянный столбец.** Никогда не overlay, не drawer
6. **Одна primary-кнопка на экран**

## Проверка готовности (e2e сценарий)

```
1. Зарегистрироваться / войти
2. Заполнить свои реквизиты
3. Создать контрагента, добавить реквизиты и подписанта
4. Создать документ → выбрать контрагента → AI-настройки
5. Рабочий экран: увидеть сгенерированный документ
6. Через AI-чат попросить внести правку → новая версия создаётся
7. Проверить документ → увидеть оценку и замечания
8. Сравнить версию 1 и версию 2
9. Утвердить версию → модалка подтверждения → баланс списывается
10. Скачать купленную версию → повторно скачать бесплатно
11. Проверить историю платежей
12. Проверить хранилище
```
